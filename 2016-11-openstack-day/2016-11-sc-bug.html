    Cluster extension

    Why Bright for cloud bursting?

  • it just works
  • seamless feel, management, and monitoring
  • cmsub (moves your data)
  • cm-scale-cluster (auto scale, DDC)
    •

    cmsub

  • User submit jobs, define in/out data
  • Bright auto transfers input to S3 (AWS)
  • Input available to HPC job
  • Job gets started... and finishes
  • Bright transfers output back to on-prem
    •

    cm-scale-cluster

  • Bright monitors job queues
  • custom policies determine if more cloud nodes are needed
  • if queue_length > x: start_nodes()
  • if queue_length < y: stop_nodes()
    •

    Recap

    • 2 types of cloud bursting
    • in Bright 7.3 it's even easier
    • cmsub + cm-scale-cluster
    • more cloud providers in the future

    COD for OpenStack Architecture

    COD for OpenStack - implementation

  • User request cluster (requirements)
  • COD tool constructs a "Heat Template"
  • Template submitted to OpenStack Heat
  • Copy-on-write from generic image (Ceph)
  • Head node customized to user's requirements
  • Head node is ready
    •
